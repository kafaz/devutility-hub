# DevUtility Agent Diagnosis 中文调用模板

以下模板用于调用 `$devutility-agent-diagnosis`。

## 1. 按节点名排查

```text
使用 $devutility-agent-diagnosis 对节点 "<节点名或别名>" 进行问题诊断。
先解析节点，再建立或复用会话；如果节点存在默认预操作模板，则先执行预操作，再执行只读且有边界的诊断命令。
重点关注 "<故障现象>"。
输出：
1. 故障现象摘要
2. 关键证据
3. 最可能根因
4. 建议的下一步命令或动作
```

## 2. 按 IP 或直连方式排查

```text
使用 $devutility-agent-diagnosis 对一台可能未注册的主机进行诊断。
如果无法解析到已注册节点，则使用直连方式建立会话：
host="<IP>"，port="<端口>"，username="<用户名>"，并使用本地可用认证方式。
仅在目标环境需要时执行预操作。
围绕 "<故障现象>" 执行安全、只读且有边界的诊断命令，并总结证据。
```

## 3. 继续已有排查

```text
使用 $devutility-agent-diagnosis 继续已有诊断会话。
先列出活动会话，找到 "<节点名或主机>" 对应的会话，保留当前 PTY 上下文，只执行确认或否定 "<待验证假设>" 所需的附加命令。
输出新增证据以及该假设是否成立。
```

## 4. 服务不可用排查

```text
使用 $devutility-agent-diagnosis 排查节点 "<节点名>" 上服务 "<服务名>" 不可用的问题。
完成预操作后，检查进程状态、端口监听、最近错误日志、系统资源，以及是否存在 OOM 或磁盘打满信号。
优先使用有边界的命令，例如 ps、ss、tail、grep、df、free、dmesg。
输出：
1. 当前故障表现
2. 最强证据
3. 最可能的问题域
4. 建议的立即行动
```

## 5. 性能劣化或响应变慢

```text
使用 $devutility-agent-diagnosis 排查节点 "<节点名>" 上的延迟升高或吞吐下降问题。
检查负载、CPU、内存、磁盘延迟、线程压力和最近应用告警，所有命令都必须是非交互且有边界的。
说明证据更像是 CPU 饱和、内存压力、磁盘 I/O、网络问题，还是下游依赖变慢。
```

## 6. 存储或日志类问题

```text
使用 $devutility-agent-diagnosis 检查节点 "<节点名>" 上的存储或日志异常。
先建立上下文，需要时切换到相关目录，然后围绕 "<路径或设备>" 执行聚焦命令。
只保留最关键的输出，并解释这些输出意味着什么。
```

## 7. 严格安全模式

```text
使用 $devutility-agent-diagnosis 在严格安全模式下排查节点 "<节点名>"。
只允许执行只读命令。
禁止安装软件、修改文件、重启服务或任何可能产生变更的操作。
如果继续排查必须提升权限或执行变更动作，停止并明确说明需要什么额外操作。
```

## 8. 适配其他 Agent 框架的稳定模板

```text
使用 $devutility-agent-diagnosis。
目标：<节点名、别名、nodeId 或主机>
目标问题：<问题描述>
约束：
- 优先使用已注册节点
- 如可复用则复用已有会话
- 若存在预操作模板，则先执行再跑 PTY 命令
- 除非明确要求，否则只运行有边界的只读命令
- 不要输出大段原始日志，优先总结证据
输出：
1. 故障现象
2. 关键证据
3. 最可能原因
4. 结论置信度
5. 下一步建议
```

## 9. 协作式人机联动排障

```text
使用 $devutility-agent-diagnosis 做一次协作式问题定位。
目标：<节点名、别名、nodeId 或主机>
问题现象：<问题描述>
执行要求：
- 先做一轮有边界的初始排查
- 如果初轮后置信度仍然不高，或者证据互相冲突，不要继续盲目扩散命令
- 改为向我输出：
  1. 已确认的证据
  2. 仍未解释的异常点
  3. 已执行过的关键命令或 run
  4. 你需要我补充的业务背景、可疑日志线索或下一步授权
- 在我回复后，复用当前会话，围绕我的提示只执行最小必要的后续命令
- 最终说明这次排查里哪些步骤、规则或提问方式值得固化到 Skill 中
```

## 10. SOP 优先排障

```text
使用 $devutility-agent-diagnosis 进行 SOP 优先排障。
目标：<节点名、别名、nodeId 或主机>
问题现象：<问题描述>
执行要求：
- 先判断是否存在可复用的 SOP 或 SOP 片段
- 如果有，优先按 SOP 的意图、变量、检查项和判定规则推进，而不是从零开始发散
- 如果 SOP 只能覆盖一部分，先跑公共部分，再把新出现的差异路径单独补充
- 结束后说明：
  1. 复用了哪个 SOP
  2. 哪个步骤或规则不够用
  3. 哪些新步骤、变量、正则或提示值得渐进式固化回 SOP
```

## 11. 最小调用模板

```text
使用 $devutility-agent-diagnosis 诊断节点 "<节点>" 上的 "<问题>"，并总结关键证据和最可能根因。
```
